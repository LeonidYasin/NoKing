## Компоненты архитектуры

### 1. Базовый слой (Blockchain Layer)

**Назначение:** Обеспечение консенсуса, безопасности и исполнения смарт-контрактов.

| Параметр | Значение |
|----------|----------|
| Сеть | Polygon / Base / Arbitrum (L2) |
| Токен | USDC / DAI (стабильные монеты) |
| Газ | Минимальный (L2) |
| Время блока | ~2 секунды |

**Почему L2:**
- Быстрое подтверждение (важно для UX).
- Быстро
- Экономия для участников.

### 2. Протокольный слой (Protocol Layer)

#### 2.1. Exchange.sol — ядро протокола

**Назначение:** Создание, управление и завершение соглашений об обмене.

**Основные функции:**
- `createAgreement()` — создать новое соглашение.
- `confirmAgreement()` — подтвердить получение (от стороны Б).
- `
- `cancelAgreement()` — отменить соглашение до исполнения.
- `disputeAgreement()` — инициировать спор.
- `resolveAgreement()` — исполнить решение арбитра.

**Хранимые данные:**
Agreement {
id: bytes32
partyA: address // кто отдаёт
partyB: address // кто получает
valueA: bytes // что отдаётся (описание)
valueB: bytes // что отдаётся взамен
amountB: uint256 // сумма в USDC (если деньги)
deadline: uint256 // срок исполнения
status: Status // PENDING, ACTIVE, COMPLETED, DISPUTED, CANCELLED
fulfilledA: bool // сторона А исполнила?
confirmedB: bool // сторона Б подтвердила?
arbitrator: address // назначенный арбитр
disputeReason: string
resolution: bytes // решение арбитра
}


---

#### 2.2. Escrow.sol — блокировка средств

**Назначение:** Безопасное удержание средств до исполнения условий.

**Основные функции:**
- `deposit()` — положить средства в эскроу.
- `release()` — отпустить средства продавцу.
- `refund()` — вернуть средства покупателю.
- `freeze()` — заморозить при споре.

**Схема работы:**
Покупатель отправляет сумму в Escrow

Средства блокируются (недоступны никому)

Продавец исполняет свои обязательства

Покупатель подтверждает получение

Escrow переводит средства продавцу

Если спор:

Любая сторона инициирует спор

Средства замораживаются

Арбитр принимает решение

Escrow исполняет решение (возврат или выплата)

text

---

#### 2.3. Rating.sol — система репутации

**Назначение:** Хранение неизменяемых рейтингов и отзывов.

**Особенности:**
- Рейтинг привязан к кошельку, а не к платформе.
- Отзывы подписываются приватным ключом.
- Невозможно удалить или подделать отзыв.

**Функции:**
- `getRating(address)` — получить рейтинг участника.
- `getRating(address)` — получить рейтинг участника
- `getReviews(address)` — получить все отзывы об участнике.

---

#### 2.4. Arbitration.sol — разрешение споров

**Назначение:** Прозрачное и справедливое разрешение споров.

**Процесс:**
1. Участник инициирует спор (с указанием причины и доказательств).
2. Система назначает арбитра (случайно из пула).
3. Арбитр изучает доказательства и выносит решение.
5. Результат фиксируется в блокчейне.
5. Результа

---

#### 2.5. DAO.sol — управление сообществом

**Назначение:** Коллективное принятие решений.

**Основные параметры:**
- Кворум: 40% активных участников.
- Пороги: 50% (обычные), 75% (принципиальные), 90% (форк).
- Один участник = один голос.

**Функции:**
- `propose()` — предложить изменение.
- `vote()` — проголосовать.
- `execute()` — исполнить принятое решение.

## Потоки данных

### Создание соглашения
User → Frontend → WalletConnect (подпись) → Exchange.sol (createAgreement) → The Graph (индексация)


### Исполнение соглашения
Party A → Frontend → Exchange.sol (fulfillAgreement) → Event → Party B (уведомление)
Party B → Frontend → Exchange.sol (confirmAgreement) → Escrow.sol (release) → Party A (средства)


### СпорDownload
User → Frontend → Exchange.sol (disputeAgreement) → Arbitration.sol → Arbitrator (уведомление)
Arbitrator → Arbitration.sol (resolve) → Escrow.sol (release/refund)

User → Frontend → DAO.sol (propose) → Voting → DAO.sol (execute) → Smart Contracts (update)
